# API Documentation Generator Skill

Scan all FastAPI route definitions in the project and generate a complete, up-to-date `ENDPOINTS.md` reference document.

## Workflow

### Step 1: Discover All Routes

1. Read all files in `infrastructure/api/routes/`
2. For each file, extract:
   - Router prefix and tags (from `APIRouter(prefix=..., tags=...)`)
   - Every route decorator: `@router.get`, `@router.post`, `@router.put`, `@router.patch`, `@router.delete`
   - HTTP method + path
   - Function name and docstring (if any)
   - Path parameters (from function signature)
   - Query parameters (from `Query()` defaults)
   - Request body schema (from the Pydantic schema type hint)
   - Response model (from `response_model=` or return type hint)
   - Status code (from `status_code=`)
   - Auth requirements (from `Depends(get_current_user)` or `Depends(require_admin)`)

### Step 2: Read Pydantic Schemas

For each request/response schema referenced by routes:

1. Read the schema file from `infrastructure/api/schemas/`
2. Extract all fields, types, validations, and examples from `model_config`
3. Note required vs optional fields
4. Note nested schemas

### Step 3: Read Exception Handlers

1. Read `infrastructure/api/exception_handlers.py`
2. Map domain exceptions to HTTP status codes
3. Extract error response format

### Step 4: Generate ENDPOINTS.md

Write `./ENDPOINTS.md` with this structure:

```markdown
# Endpoint Reference

> Auto-generated by `/api-doc` skill. Last updated: {current_date}

## Base URL

- Development: `http://localhost:8000`

## Authentication

All endpoints except health check and login require authentication via JWT Bearer token.

| Header | Value |
|--------|-------|
| `Authorization` | `Bearer <access_token>` |

### Auth Levels

| Level | Description |
|-------|-------------|
| Public | No authentication required |
| Authenticated | Valid JWT token required |
| Admin | JWT token with admin role required |

---

## Endpoints

### {Tag Group} (e.g., "Judges", "Authentication")

#### `{METHOD} {full_path}`

**Auth**: {Public | Authenticated | Admin}

**Description**: {from docstring or inferred from function name}

**Path Parameters**:
| Name | Type | Description |
|------|------|-------------|

**Query Parameters**:
| Name | Type | Default | Description |
|------|------|---------|-------------|

**Request Body** (`{SchemaName}`):
```json
{example from model_config or constructed from fields}
```

**Response** `{status_code}` (`{ResponseSchemaName}`):

```json
{example response}
```

**Error Responses**:

| Status | Error Code | Description |
|--------|-----------|-------------|

---

```

### Step 5: Generate Summary Table

At the top of the document, include a quick-reference table:

```markdown
## Quick Reference

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | /health | Public | Health check |
| POST | /api/v1/auth/login | Public | User login |
| ... | ... | ... | ... |
```

### Step 6: Verify Completeness

1. Count total routes found vs routes documented
2. Flag any routes missing docstrings or response models
3. Report the results to the user

## Important Rules

- Read actual code — do NOT guess or assume endpoints exist
- Include the health check endpoint
- Group endpoints by their router tags
- Sort within each group: GET before POST before PUT before PATCH before DELETE
- Use realistic example values in JSON (not "string" or 0)
- If an endpoint has pagination, document the `skip`/`limit` query params
- Include the `Content-Type: application/json` header note for POST/PUT/PATCH
- Do NOT include internal/private routes (prefixed with `_`)
